# AI Smart Home เพื่อนคุยได้ (ESP32-S3 + Local LLM)

## 1. ชื่อโครงการ
AI Smart Home Assistant ระบบบ้านอัจฉริยะผู้ช่วยภาษาไทยที่สามารถพูดคุยโต้ตอบ ควบคุมอุปกรณ์ IoT และทำงานอัตโนมัติจากข้อมูลเซนเซอร์ โดยใช้โมเดล AI ภายในเครื่อง (Local LLM)

---

## 2. แนวคิดโครงการ
โครงการนี้พัฒนาระบบบ้านอัจฉริยะที่ยกระดับจากการสั่งงานแบบเดิม ไปสู่การมีผู้ช่วย AI ที่สามารถสนทนาภาษาไทยแบบธรรมชาติ เข้าใจบริบทของผู้ใช้ วิเคราะห์สถานการณ์จากเซนเซอร์ และควบคุมอุปกรณ์ต่าง ๆ ผ่านบอร์ด ESP32-S3 ได้แบบเรียลไทม์

ตัวอย่าง:
- ผู้ใช้: ร้อนมากเลย
- AI: วันนี้ร้อนจริง เดี๋ยวเปิดพัดลมให้ไหม
- ผู้ใช้: เปิดเลย
- ระบบสั่ง Relay เปิดพัดลมทันที

---

## 3. วัตถุประสงค์
1. พัฒนาระบบ Smart Home ที่ควบคุมอุปกรณ์ผ่าน ESP32-S3
2. ใช้ AI ภาษาไทยเพื่อสนทนาและรับคำสั่งธรรมชาติ
3. ใช้ข้อมูลเซนเซอร์ในการตัดสินใจอัตโนมัติ
4. สร้างระบบ Local AI ลดการพึ่งพา Cloud
5. รองรับการขยายระบบในอนาคต เช่น API ภายนอก / Voice Control เต็มรูปแบบ

---

## 4. อุปกรณ์ที่ใช้

### 4.1 บอร์ดหลัก
- ESP32-S3 WROOM (N16R8) แบบ USB Type-C
- จุดเด่น: RAM/Flash สูง, รองรับ AIoT, USB สะดวก

### 4.2 ระบบเสียง
- MAX98357A (I2S DAC Amplifier)
- ลำโพง 4 Ohm 3W
- INMP441 (I2S Microphone) สำหรับ Phase 2

### 4.3 เซนเซอร์และควบคุมอุปกรณ์
- DHT22 วัดอุณหภูมิ/ความชื้น
- Relay Module 5V (2 หรือ 4 Channel)
- PIR Motion Sensor HC-SR501 ตรวจจับการเคลื่อนไหว

### 4.4 อุปกรณ์ประกอบ
- Breadboard 400/830 จุด
- Jumper Wire M-M / M-F / F-F
- USB Type-C Data Cable

## 5. ซอฟต์แวร์และโมเดล AI

### 5.0 แนวคิดการทำงานแบบ Hybrid
ระบบจะไม่จำกัดแค่การควบคุมอุปกรณ์ IoT ภายในบ้าน แต่สามารถตอบคำถามทั่วไปเกี่ยวกับโลกภายนอกได้ด้วย โดยแยกการทำงานเป็น 2 ส่วน:
- **ข้อมูลภายในบ้าน (Local / IoT):** ใช้ข้อมูลจาก ESP32-S3, เซนเซอร์, และสถานะอุปกรณ์จริง
- **ข้อมูลภายนอก (Online / API):** ใช้ API ภายนอกสำหรับข้อมูลสด เช่น อากาศ ฝน การจราจร ข่าว และข้อมูลเว็บล่าสุด

แนวคิดคือให้ **Typhoon 2 8B** เป็นบุคลิกหลักของผู้ช่วย คอยเรียบเรียงคำตอบภาษาไทยแบบเพื่อนคุยกัน ส่วนข้อมูลสดให้ดึงจาก API ก่อนแล้วค่อยสรุปคำตอบออกมา

### 5.1 ฝั่ง Embedded
- MicroPython บน ESP32-S3

### 5.2 ฝั่ง Server Local
- Python 3.x
- FastAPI
- llama.cpp / Ollama
- SQLite (บันทึกประวัติ)

### 5.3 AI Models ที่ติดตั้งแล้ว
- Typhoon 2 8B (หลัก)
- Gemma 4 2B
- Gemma 4 4B

### 5.4 ทางเลือกเสริม
- เชื่อม API Key ภายนอกได้ภายหลัง เช่น OpenAI / Gemini / Claude

## 6. สถาปัตยกรรมระบบ

ระบบถูกออกแบบแบบ **Hybrid AI + IoT Architecture** เพื่อให้รองรับทั้งงานควบคุมอุปกรณ์ภายในบ้าน งานสนทนาทั่วไป และงานดึงข้อมูลสดจากภายนอก โดยมีหลักการสำคัญคือ
- **ESP32-S3** รับผิดชอบงานอุปกรณ์, เซนเซอร์, ลำโพง, ไมโครโฟน และการควบคุมรีเลย์
- **Local Server** เป็นสมองกลางของระบบ ทำหน้าที่รับคำถาม, เรียก AI, เรียก API ภายนอก, และส่งคำสั่งกลับไปยัง ESP32
- **Typhoon 2 8B** เป็นโมเดลหลักสำหรับการคุยภาษาไทย, สรุปคำตอบ, และสร้างบุคลิกของผู้ช่วย
- **External APIs** เป็นแหล่งข้อมูลสด เช่น อากาศ ฝน การจราจร ข่าว และข้อมูลล่าสุด

### 6.1 ภาพรวมสถาปัตยกรรม

```text
[ผู้ใช้]
   ↓ พูด / พิมพ์
[Web UI / Voice Interface]
   ↓
[FastAPI Local Server]
   ├─ Intent Router
   ├─ Conversation Manager
   ├─ Device Control Engine
   ├─ Sensor Data Manager
   ├─ External API Manager
   ├─ TTS/STT Manager
   └─ Logging / Memory / Database
          ↓
   [Typhoon 2 8B]
          ↓
   [ESP32-S3]
      ├─ DHT22
      ├─ PIR Motion Sensor
      ├─ Relay Module
      ├─ MAX98357A + Speaker
      └─ INMP441 Microphone
```

### 6.2 แยกบทบาทของแต่ละส่วน

#### 6.2.1 ESP32-S3 Layer
หน้าที่:
- เชื่อมต่อ Wi-Fi และสื่อสารกับ Server
- อ่านค่าจาก DHT22 และ PIR
- ควบคุม Relay เปิด/ปิดอุปกรณ์
- ส่งข้อมูลเซนเซอร์ขึ้น Server เป็นระยะ
- เล่นเสียงตอบกลับผ่าน MAX98357A + Speaker
- รองรับรับเสียงจาก INMP441 สำหรับการพัฒนาระยะถัดไป

#### 6.2.2 Local Server Layer
หน้าที่:
- รับคำถามจากผู้ใช้ผ่าน UI หรือเสียง
- วิเคราะห์ว่าเป็นคำถามประเภทใด
- เรียกโมเดล Typhoon 2 8B เพื่อสร้างคำตอบ
- เรียก API ภายนอกถ้าต้องใช้ข้อมูลสด
- ตัดสินใจว่าจะสั่ง ESP32 หรือไม่
- จัดเก็บ history, logs, และ memory ของบทสนทนา

#### 6.2.3 External Service Layer
หน้าที่:
- ให้ข้อมูลที่อัปเดตแบบเรียลไทม์
- ใช้เมื่อคำถามเกี่ยวกับอากาศ ฝน จราจร ข่าว หรือข้อมูลล่าสุด
- ใช้เป็น fallback เมื่อต้องการความรู้ที่อยู่นอกขอบเขตของ local model

## 6.1 ความสามารถนอกเหนือจาก IoT

ระบบนี้สามารถทำหน้าที่เป็น **AI ผู้ช่วยทั่วไป** ได้ ไม่ได้จำกัดแค่การสั่งเปิดปิดอุปกรณ์ภายในบ้าน โดยใช้วิธีดังนี้:

### 6.1.1 ถามตอบเรื่องสภาพอากาศและฝน
ตัวอย่างคำถาม:
- ข้างนอกร้อนไหม
- วันนี้ฝนจะตกไหม
- ยะลาตอนเย็นต้องพกร่มไหม

แนวทางทำงาน:
1. Server รับคำถามจากผู้ใช้
2. ตรวจว่าเป็นคำถามด้านอากาศ
3. เรียก **OpenWeather API** เพื่อดึงข้อมูลอากาศปัจจุบันและพยากรณ์
4. ส่งผลลัพธ์ให้ Typhoon 2 8B เรียบเรียงเป็นภาษาไทยธรรมชาติ

ข้อมูลที่ใช้ได้:
- อุณหภูมิ
- ความชื้น
- โอกาสฝนตก
- สภาพฟ้า
- พยากรณ์รายชั่วโมง/รายวัน
- การแจ้งเตือนสภาพอากาศ

อ้างอิง: OpenWeather ระบุว่ามี API สำหรับ current weather, forecast, historical data, air pollution และ One Call API 3.0 ที่รวม current, forecast และ alerts ไว้ใน endpoint เดียว

### 6.1.2 ถามตอบเรื่องการจราจร
ตัวอย่างคำถาม:
- ยะลารถติดไหม
- ตอนนี้ถนนหน้า ม. รถเยอะไหม
- ถ้าจะไปตลาดตอนนี้ใช้เวลาประมาณกี่นาที

แนวทางทำงาน:
1. Server รับคำถาม
2. แปลงชื่อสถานที่เป็นพิกัด (Geocoding)
3. เรียก **Google Maps Platform Routes API** หรือบริการ traffic-aware routes
4. ให้ Typhoon 2 8B สรุปเป็นภาษาคน

ข้อมูลที่ใช้ได้:
- เวลาเดินทางโดยประมาณ
- เส้นทางที่รถติดน้อยกว่า
- สถานะการจราจรโดยอิง route traffic
- เปรียบเทียบเวลาเดินทางหลายเส้นทาง

อ้างอิง: Google Maps Platform Routes API รองรับการคำนวณเส้นทางแบบ traffic-aware และกำหนด routing preference เช่น TRAFFIC_AWARE หรือ TRAFFIC_AWARE_OPTIMAL ได้

### 6.1.3 ถามตอบข่าวหรือข้อมูลล่าสุด
ตัวอย่างคำถาม:
- วันนี้มีข่าวอะไรบ้าง
- ข่าวเทคโนโลยีล่าสุดคืออะไร
- ราคาทองวันนี้ประมาณเท่าไหร่

แนวทางทำงาน:
1. Server รับคำถาม
2. ถ้าเป็นข่าว ใช้ **NewsAPI** หรือ web search API
3. ถ้าเป็นข้อมูลสดเฉพาะด้าน อาจใช้ API ตามหมวดนั้นเพิ่มเติม
4. ให้ Typhoon 2 8B สรุปแบบสั้น อ่านง่าย

### 6.1.4 ถามตอบความรู้ทั่วไป
ตัวอย่างคำถาม:
- อธิบาย IoT คืออะไร
- ทำไมฝนถึงตก
- ช่วยสรุปเรื่อง MicroPython ให้หน่อย

แนวทางทำงาน:
- ถ้าเป็นความรู้ทั่วไปที่ไม่ต้องอัปเดตมาก ใช้ Typhoon 2 8B ตอบได้โดยตรง
- ถ้าต้องการข้อมูลล่าสุดหรือแม่นขึ้น ให้ใช้ web search API เพิ่ม แล้วสรุปอีกชั้นหนึ่ง

### 6.1.5 โหมดคุยเล่นเหมือนเพื่อน
ตัวอย่างคำถาม:
- วันนี้เหนื่อยจัง
- มีมุกอะไรเล่นไหม
- แนะนำอะไรให้ทำตอนเย็นหน่อย

แนวทางทำงาน:
- ใช้ Typhoon 2 8B เป็นโมเดลหลักในการคุย
- ตั้ง system prompt ให้พูดแบบเพื่อนสนิท
- ถ้าคำถามไม่ต้องใช้ข้อมูลสด ก็ไม่ต้องเรียก API ภายนอก

## 6.2 API ที่ควรใช้

### 6.2.1 OpenWeather API
ใช้สำหรับ:
- อากาศปัจจุบัน
- พยากรณ์ฝน
- พยากรณ์รายชั่วโมง
- weather alerts

เหตุผลที่แนะนำ:
- เริ่มใช้งานง่าย
- มี current weather และ forecast ครบ
- มี One Call API 3.0 สำหรับรวมข้อมูล current, forecast, historical และ alerts

สิ่งที่ต้องมี:
- API key
- พิกัดหรือชื่อเมือง

ตัวอย่าง use case:
- ข้างนอกร้อนไหม
- ฝนจะตกไหม
- วันนี้ที่ยะลาอากาศเป็นยังไง

### 6.2.2 Google Maps Platform API
ใช้สำหรับ:
- Geocoding แปลงชื่อสถานที่เป็นพิกัด
- Routes API สำหรับคำนวณเส้นทาง
- Traffic-aware route estimation

เหตุผลที่แนะนำ:
- รองรับข้อมูล route และ ETA แบบอิงการจราจร
- ใช้ต่อยอดกับ dashboard แผนที่ได้

สิ่งที่ต้องมี:
- Google Cloud project
- เปิดใช้งาน Geocoding API / Routes API
- API key

ตัวอย่าง use case:
- ยะลารถติดไหม
- จากบ้านไปมหาวิทยาลัยใช้เวลากี่นาที
- เส้นไหนรถน้อยกว่ากัน

### 6.2.3 NewsAPI
ใช้สำหรับ:
- ข่าวล่าสุด
- ข่าวตามหมวด เช่น เทคโนโลยี ธุรกิจ กีฬา
- สรุปหัวข้อข่าววันนี้

เหตุผลที่แนะนำ:
- ใช้งานง่าย
- เหมาะกับดึง headline และบทความล่าสุด

สิ่งที่ต้องมี:
- API key

ตัวอย่าง use case:
- วันนี้มีข่าวอะไรบ้าง
- ข่าว AI ล่าสุดคืออะไร

### 6.2.4 Web Search API (เช่น Tavily หรือบริการค้นเว็บอื่น)
ใช้สำหรับ:
- คำถามเปิดกว้างที่ไม่รู้ว่าจะไปดึงจาก API ไหน
- คำถามที่ต้องการข้อมูลล่าสุดจากเว็บ
- คำถามแนวอธิบายพร้อมอ้างอิง

เหตุผลที่แนะนำ:
- ยืดหยุ่นสูง
- เหมาะกับ AI assistant แบบถามได้ทุกเรื่อง

ตัวอย่าง use case:
- ตอนนี้มีเหตุการณ์อะไรสำคัญบ้าง
- ช่วยหาข้อมูลล่าสุดเรื่องพายุเข้าไทย
- มหาวิทยาลัยเปิดเทอมวันไหน

### 6.2.5 ทางเลือก API Key ภายนอกสำหรับ LLM
หากในอนาคตต้องการความสามารถสูงขึ้นหรืออยากเปรียบเทียบผลลัพธ์ สามารถเพิ่ม cloud LLM ได้ง่าย เช่น:
- OpenAI API
- Google Gemini API
- Anthropic Claude API

แนวทางใช้งาน:
- ใช้ Typhoon 2 8B เป็นตัวหลักแบบ local
- ถ้าคำถามยากมากหรืออยากได้ fallback ให้ส่งต่อไป cloud model ได้
- กำหนด policy ว่าเมื่อไรควรใช้ local / เมื่อไรควรใช้ cloud

## 6.3 Logic การเลือกแหล่งข้อมูล

ระบบควรมีตัวกลางสำหรับจำแนกคำถามก่อนส่งต่อ เช่น:

1. **ถ้าเป็นคำถามในบ้าน** → ใช้ข้อมูลจากเซนเซอร์/ESP32
2. **ถ้าเป็นคำถามข้อมูลสด** → เรียก API ภายนอก
3. **ถ้าเป็นคำถามความรู้ทั่วไป** → ให้ Typhoon ตอบตรง
4. **ถ้าเป็นคำถามล่าสุด/เฉพาะทางมาก** → ใช้ web search หรือ cloud API เพิ่ม

ตัวอย่าง:
- ร้อนในห้องไหม → DHT22
- ข้างนอกร้อนไหม → OpenWeather API
- ยะลารถติดไหม → Google Maps Routes API
- อธิบาย IoT คืออะไร → Typhoon 2 8B
- ข่าว AI วันนี้ → NewsAPI หรือ Web Search API

## 7. ความสามารถของระบบ

### 7.1 สนทนาภาษาไทยเหมือนเพื่อน
- ตอบโต้แบบธรรมชาติ
- จำบริบทสนทนา
- ใช้ภาษากันเองได้

### 7.2 ควบคุม Smart Home
- เปิด/ปิดไฟ
- เปิด/ปิดพัดลม
- เปิด/ปิดปลั๊กไฟ
- ควบคุมหลายอุปกรณ์พร้อมกัน

### 7.3 เข้าใจคำสั่งธรรมชาติ
- มืดจัง → เปิดไฟ
- ร้อนมาก → เปิดพัดลม
- จะนอนแล้ว → ปิดไฟบางส่วน

### 7.4 ใช้เซนเซอร์ตัดสินใจ
- อุณหภูมิสูง → แจ้งเตือน/เปิดพัดลม
- ตรวจพบคนเดินผ่าน → AI ทักทาย

## 7.1 Data Flow การทำงานของระบบ

### 7.1.1 Flow แบบควบคุม IoT
```text
ผู้ใช้พูด/พิมพ์คำสั่ง
   ↓
Server รับข้อความ
   ↓
Intent Router วิเคราะห์ว่าเป็นคำสั่งควบคุมอุปกรณ์
   ↓
Device Control Engine ตรวจสอบสถานะอุปกรณ์และความเป็นไปได้
   ↓
ส่งคำสั่งไป ESP32-S3
   ↓
ESP32 ควบคุม Relay
   ↓
Server รับผลลัพธ์กลับมา
   ↓
Typhoon สร้างข้อความตอบกลับแบบเป็นธรรมชาติ
   ↓
ตอบกลับเป็นข้อความ/เสียง
```

### 7.1.2 Flow แบบถามข้อมูลในบ้าน
```text
ผู้ใช้ถาม เช่น ห้องร้อนไหม
   ↓
Server ตรวจว่าเป็นข้อมูลจากเซนเซอร์
   ↓
ดึงข้อมูลล่าสุดจาก Sensor Data Manager
   ↓
ส่งข้อมูล context ให้ Typhoon
   ↓
Typhoon สรุปเป็นภาษาไทยแบบเพื่อนคุยกัน
   ↓
ตอบกลับข้อความ/เสียง
```

### 7.1.3 Flow แบบถามข้อมูลภายนอก
```text
ผู้ใช้ถาม เช่น ข้างนอกร้อนไหม / ฝนจะตกไหม / ยะลารถติดไหม
   ↓
Server ตรวจว่าเป็นคำถามข้อมูลสด
   ↓
External API Manager เรียก API ที่เหมาะสม
   ↓
ได้ข้อมูล JSON / structured data กลับมา
   ↓
ส่งข้อมูลให้ Typhoon เรียบเรียงคำตอบ
   ↓
ตอบกลับข้อความ/เสียง
```

### 7.1.4 Flow แบบคุยทั่วไป
```text
ผู้ใช้ถามคำถามทั่วไป
   ↓
Conversation Manager รับ context การคุยก่อนหน้า
   ↓
ส่ง prompt + memory ให้ Typhoon 2 8B
   ↓
Typhoon สร้างคำตอบ
   ↓
TTS แปลงเป็นเสียง (ถ้าเปิด voice mode)
   ↓
ตอบกลับผู้ใช้
```

## 7.2 Decision Flow การเลือกแหล่งข้อมูล

ระบบควรมีตัวกลางในการจำแนกเจตนา (Intent Router) โดยใช้หลักการดังนี้:

### ประเภทคำถาม
1. **Device Control Intent**
   - เช่น เปิดไฟ ปิดพัดลม เปิดปลั๊ก
   - ส่งต่อไปยัง Device Control Engine

2. **Sensor Query Intent**
   - เช่น ตอนนี้ห้องร้อนไหม ความชื้นเท่าไหร่
   - ดึงข้อมูลจาก Sensor Data Manager

3. **Realtime External Info Intent**
   - เช่น ฝนจะตกไหม ยะลารถติดไหม ข่าววันนี้มีอะไรบ้าง
   - เรียก External APIs

4. **General Knowledge Intent**
   - เช่น IoT คืออะไร ทำไมท้องฟ้าสีฟ้า
   - ให้ Typhoon ตอบตรง

5. **Friendly Chat Intent**
   - เช่น วันนี้เหนื่อยจัง คิดถึงจัง ช่วยคุยหน่อย
   - ใช้ Typhoon พร้อม personality prompt

### ตัวอย่าง Logic
```text
if message เกี่ยวกับคำสั่งอุปกรณ์:
    route -> Device Control Engine
elif message เกี่ยวกับอุณหภูมิ/แสง/คนในบ้าน:
    route -> Sensor Data Manager
elif message เกี่ยวกับอากาศ/จราจร/ข่าว/ข้อมูลล่าสุด:
    route -> External API Manager
else:
    route -> Typhoon General Chat
```

## 7.3 โครงสร้างโมดูลของ Server

เพื่อให้ระบบเสถียรและขยายต่อได้ ควรแยกโมดูลของ Server ดังนี้:

### 7.3.1 API Gateway / FastAPI Layer
หน้าที่:
- รับ request จาก Web UI / Mobile UI / ESP32
- ส่งต่อไปยัง service ภายใน
- จัดการ response format

ตัวอย่าง endpoint:
- `POST /chat`
- `POST /device/control`
- `GET /sensors/latest`
- `GET /devices/status`
- `POST /voice/transcribe`
- `POST /voice/speak`

### 7.3.2 Intent Router
หน้าที่:
- จำแนกประเภทคำถาม
- ส่งงานไปยังโมดูลที่ถูกต้อง

แนวทางพัฒนา:
- เริ่มจาก rule-based + keyword
- ภายหลังค่อยใช้ lightweight classifier หรือ LLM function calling

### 7.3.3 Conversation Manager
หน้าที่:
- เก็บ context การสนทนา
- แนบ system prompt
- จำกัดจำนวน history เพื่อให้ inference เร็ว
- ควบคุม style การตอบให้เหมือนเพื่อน

### 7.3.4 LLM Manager
หน้าที่:
- โหลด Typhoon 2 8B
- เรียก inference
- จัดการ streaming response
- ตั้งค่าความเร็ว เช่น max tokens, temperature, top-p

### 7.3.5 External API Manager
หน้าที่:
- เชื่อม OpenWeather
- เชื่อม Google Maps / Traffic
- เชื่อม NewsAPI / Web Search API
- ทำ normalization ให้ข้อมูลอยู่ในรูปแบบเดียวกัน

### 7.3.6 Device Control Engine
หน้าที่:
- รับคำสั่งจาก Intent Router
- ตรวจสอบว่าอุปกรณ์ใดถูกควบคุมได้
- ส่งคำสั่งไปยัง ESP32
- บันทึกสถานะและ log

### 7.3.7 Sensor Data Manager
หน้าที่:
- รับค่าจาก DHT22 / PIR
- เก็บค่าล่าสุด
- สรุป context เพื่อส่งต่อให้ LLM

### 7.3.8 Voice Manager
หน้าที่:
- Speech-to-Text สำหรับรับเสียง
- Text-to-Speech สำหรับตอบกลับด้วยเสียง
- ส่งไฟล์เสียงไปยังลำโพงหรือ UI

### 7.3.9 Database / Memory Layer
หน้าที่:
- บันทึกประวัติการคุย
- บันทึก log การควบคุมอุปกรณ์
- เก็บ preferences ของผู้ใช้ เช่น ชอบให้ AI ตอบสั้น ตอบแบบกันเอง

แนะนำเบื้องต้น:
- SQLite สำหรับเวอร์ชันเริ่มต้น
- ภายหลังค่อยขยายเป็น PostgreSQL หากระบบใหญ่ขึ้น

## 12.1 Wiring และการเชื่อมต่ออุปกรณ์

> หมายเหตุ: ก่อนต่อวงจรจริงควรตรวจสอบแรงดันไฟและ pinout ของบอร์ด ESP32-S3 รุ่นที่ใช้อีกครั้งจาก datasheet ของผู้ผลิต เนื่องจากบางบอร์ดมีการจัดขาไม่เหมือนกัน

### 12.1.1 การเชื่อมต่อ DHT22
อุปกรณ์:
- VCC
- DATA
- GND

แนวทางต่อ:
- VCC → 3.3V
- DATA → GPIO ที่กำหนดสำหรับอ่านข้อมูล
- GND → GND
- ใส่ตัวต้านทาน pull-up 4.7k–10k ระหว่าง VCC กับ DATA หากโมดูลที่ซื้อมาไม่มีในตัว

หน้าที่:
- อ่านค่าอุณหภูมิและความชื้น
- ส่งข้อมูลขึ้น Server ทุกช่วงเวลา เช่น ทุก 5–10 วินาที

### 12.1.2 การเชื่อมต่อ PIR Motion Sensor (HC-SR501)
อุปกรณ์:
- VCC
- OUT
- GND

แนวทางต่อ:
- VCC → 5V หรือ 3.3V ตามสเปกของโมดูล
- OUT → GPIO input
- GND → GND

หน้าที่:
- ตรวจจับการเคลื่อนไหว
- Trigger ให้ระบบทักทายอัตโนมัติ หรือใช้เป็นเงื่อนไข automation

### 12.1.3 การเชื่อมต่อ Relay Module 5V
อุปกรณ์:
- VCC
- GND
- IN1 / IN2 / IN3 / IN4

แนวทางต่อ:
- VCC → 5V
- GND → GND ร่วมกับระบบ
- IN1/IN2 → GPIO output

หน้าที่:
- ควบคุมการเปิดปิดอุปกรณ์จำลอง เช่น พัดลม ไฟ หรือปลั๊ก

ข้อควรระวัง:
- ถ้าใช้กับไฟบ้านจริงต้องระวังความปลอดภัยสูงมาก
- สำหรับเวอร์ชันโปรเจ็กต์เริ่มต้น แนะนำใช้หลอดไฟ DC หรืออุปกรณ์ low voltage เพื่อสาธิตก่อน

### 12.1.4 การเชื่อมต่อ MAX98357A + Speaker
อุปกรณ์:
- VIN
- GND
- LRC / WS
- BCLK
- DIN
- Speaker + / -

แนวทางต่อ:
- VIN → 5V หรือ 3.3V ตามโมดูล
- GND → GND
- LRC / WS → GPIO I2S WS
- BCLK → GPIO I2S BCLK
- DIN → GPIO I2S DATA OUT
- SPK+ / SPK- → ลำโพง 4 Ohm 3W

หน้าที่:
- รับข้อมูลเสียงจาก ESP32-S3
- ขับลำโพงให้ระบบพูดตอบกลับ

### 12.1.5 การเชื่อมต่อ INMP441 (Phase 2)
อุปกรณ์:
- VDD
- GND
- SCK
- WS
- SD
- L/R

แนวทางต่อ:
- VDD → 3.3V
- GND → GND
- SCK → GPIO I2S BCLK
- WS → GPIO I2S WS
- SD → GPIO I2S DATA IN
- L/R → กำหนด channel ตามต้องการ

หน้าที่:
- รับเสียงจากผู้ใช้
- ส่งข้อมูลไป Server เพื่อทำ Speech-to-Text

### 12.1.6 ตัวอย่างกลุ่ม GPIO ที่ควรกำหนดในโค้ด
> ตัวเลข GPIO จริงควรยืนยันตามบอร์ด ESP32-S3 ที่ใช้

ตัวอย่างแนวคิด:
- DHT22_DATA_PIN
- PIR_INPUT_PIN
- RELAY_1_PIN
- RELAY_2_PIN
- I2S_BCLK_PIN
- I2S_WS_PIN
- I2S_DOUT_PIN
- I2S_DIN_PIN

ควรแยกเป็นไฟล์ config เพื่อเปลี่ยน pin ได้ง่าย

## 12.2 API Endpoints ที่ควรมี

### 12.2.1 ฝั่ง Server สำหรับ UI / Client

#### `POST /chat`
หน้าที่:
- รับข้อความจากผู้ใช้
- วิเคราะห์ intent
- เลือกแหล่งข้อมูล
- ส่งคำตอบกลับ

ตัวอย่าง request:
```json
{
  "message": "ข้างนอกร้อนไหม",
  "mode": "text"
}
```

ตัวอย่าง response:
```json
{
  "reply": "ตอนนี้ที่ยะลาอากาศค่อนข้างร้อน ประมาณ 33 องศา และมีเมฆบางส่วน",
  "intent": "weather_query",
  "source": "openweather"
}
```

#### `POST /voice/chat`
หน้าที่:
- รับไฟล์เสียงหรือสตรีมเสียงจาก client
- แปลงเสียงเป็นข้อความ
- ส่งเข้า logic เดียวกับ `/chat`
- ส่งกลับข้อความและ/หรือไฟล์เสียงตอบกลับ

#### `GET /devices/status`
หน้าที่:
- คืนสถานะอุปกรณ์ทั้งหมด เช่น relay แต่ละช่องเปิดหรือปิด

#### `GET /sensors/latest`
หน้าที่:
- คืนค่าล่าสุดของ DHT22, PIR และเซนเซอร์อื่น

#### `GET /logs/conversation`
หน้าที่:
- ดูประวัติการสนทนา

#### `GET /logs/device-actions`
หน้าที่:
- ดู log การควบคุมอุปกรณ์

---

### 12.2.2 ฝั่ง Server สำหรับ ESP32

#### `POST /esp32/sensor`
หน้าที่:
- ESP32 ส่งค่าเซนเซอร์ขึ้นมาเก็บที่ Server

ตัวอย่าง request:
```json
{
  "temperature": 31.8,
  "humidity": 74.2,
  "motion": true,
  "timestamp": "2026-04-22T14:20:00"
}
```

#### `GET /esp32/commands`
หน้าที่:
- ESP32 polling เพื่อรับคำสั่งล่าสุดจาก Server
- เช่น เปิด relay, เล่นเสียง, รีเฟรชสถานะ

#### `POST /esp32/status`
หน้าที่:
- ESP32 รายงานผลการทำงาน เช่น relay เปิดสำเร็จหรือไม่

#### `POST /esp32/heartbeat`
หน้าที่:
- แจ้งว่าอุปกรณ์ยัง online อยู่

## 12.4 Roadmap การพัฒนา

### Phase 1: Core IoT + Chat
เป้าหมาย:
- ESP32 ต่อ Wi-Fi ได้
- อ่าน DHT22 ได้
- ควบคุม relay ได้
- Server มี `/chat` และ `/esp32/sensor`
- Typhoon ตอบข้อความภาษาไทยได้

สิ่งที่ควรเดโมได้:
- ผู้ใช้พิมพ์ว่า “เปิดพัดลม” แล้ว relay ทำงาน
- ผู้ใช้ถามว่า “ห้องร้อนไหม” แล้ว AI ตอบจากเซนเซอร์

### Phase 2: Voice Output + Friendly Chat
เป้าหมาย:
- เพิ่ม TTS ให้ระบบพูดตอบผ่านลำโพง
- ปรับ personality ให้เหมือนเพื่อน
- เพิ่ม memory ของบทสนทนา

สิ่งที่ควรเดโมได้:
- ผู้ใช้พิมพ์ถาม แล้วระบบตอบออกเสียง
- ผู้ใช้คุยทั่วไปได้ เช่น “วันนี้เหนื่อยจัง”

### Phase 3: External APIs
เป้าหมาย:
- เชื่อม OpenWeather
- เชื่อม Google Maps / Traffic
- เชื่อม NewsAPI หรือ Web Search

สิ่งที่ควรเดโมได้:
- “ข้างนอกร้อนไหม”
- “ฝนจะตกไหม”
- “ยะลารถติดไหม”
- “วันนี้มีข่าวอะไรบ้าง”

### Phase 4: Voice Input เต็มรูปแบบ
เป้าหมาย:
- ใช้ INMP441 รับเสียง
- ทำ STT
- พูดคุยสองทางแบบใกล้เคียงผู้ช่วยอัจฉริยะ

สิ่งที่ควรเดโมได้:
- พูดเข้าไมค์แล้วระบบตอบกลับด้วยเสียง

## 12.5 แผนการพัฒนาแบบใช้งานจริง

### สัปดาห์ที่ 1
- เตรียมอุปกรณ์
- ทดสอบ ESP32-S3 + MicroPython
- ทดสอบ DHT22, relay, PIR ทีละตัว

### สัปดาห์ที่ 2
- สร้าง FastAPI server เบื้องต้น
- ทำ endpoint รับค่าจาก ESP32
- ทำ endpoint แชตเชื่อม Typhoon

### สัปดาห์ที่ 3
- เชื่อมระบบแชตกับคำสั่งควบคุมอุปกรณ์
- ทำ log และ dashboard ขั้นต้น
- ปรับ prompt ให้คุยเหมือนเพื่อน

### สัปดาห์ที่ 4
- เพิ่ม TTS
- ปรับ latency และ optimize ตอน present
- เพิ่ม weather API

### สัปดาห์ที่ 5
- เพิ่ม traffic/news/web search
- ปรับความเสถียร
- ทดสอบ scenario นำเสนอจริง

## 12.6 ข้อเสนอแนะสำหรับการพรีเซนต์

1. เตรียม demo script ล่วงหน้า 4–5 กรณี
2. warm-up Typhoon ก่อนเริ่ม
3. เปิดเฉพาะบริการที่จำเป็น
4. มี fallback mode หาก API ภายนอกล่ม
5. แยกคำถาม demo เป็นหมวด:
   - ควบคุมบ้าน
   - ถามข้อมูลในบ้าน
   - ถามข้อมูลนอกบ้าน
   - คุยเล่นกับ AI
6. ให้ระบบตอบทั้งข้อความและเสียง จะดูสมบูรณ์ที่สุด
